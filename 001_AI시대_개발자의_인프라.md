# AI 시대, 개발자의 인프라

## 개요

인프라 이해는 운영 조직만의 역량이 아니라 개발자의 기본 소양이 되었다. DevOps와 클라우드 네이티브 전환이 개발과 운영의 경계를 허물었고, 그 결과 쿠버네티스가 클라우드 인프라의 공통 언어로 부상하였다. 본 장은 여기에 AI 에이전트가 결합된 GitAIOps와, 이를 안전하게 뒷받침하는 가드레일 체계를 다룬다.

---

## DevOps와 클라우드 네이티브

전통적 조직은 개발과 운영을 분리하여 환경 불일치라는 구조적 문제를 안았다. **DevOps**는 이를 해소하는 실천 체계로, 자동화 파이프라인(CI/CD), 코드형 인프라(IaC), 관측 가능성(Observability)을 핵심으로 한다.

이 문화의 성숙은 **클라우드 네이티브** 전환으로 이어졌다. 애플리케이션과 실행 환경을 단일 이미지로 묶는 **컨테이너**, 이를 독립 배포 단위로 분할하는 **마이크로서비스**, 다수의 컨테이너를 자동 배치·복구하는 **오케스트레이션**의 결합이다. 컨테이너가 배포 단위를 표준화하였다면, 오케스트레이터는 그 운영을 자동화한다. 그 사실상 표준이 쿠버네티스이다.

---

## 쿠버네티스: 클라우드 인프라의 공통 언어

쿠버네티스(Kubernetes, k8s)의 동작은 두 개념으로 설명된다. **선언형(Declarative) 모델**에서 사용자는 명령이 아니라 원하는 상태(desired state)를 기술하고, 도달 절차는 시스템에 위임한다. **조정 루프(Reconciliation Loop)**는 원하는 상태와 현재 상태를 지속 비교하여 차이를 자동 복원한다. 이 자기 치유(self-healing)가 제어의 핵심이다. 또한 모든 리소스가 API 서버를 통해 관리되므로, 클라우드 제공자나 온프레미스와 무관하게 동일한 인터페이스로 인프라를 다룰 수 있다.

| 개념 | 역할 |
| --- | --- |
| Pod | 컨테이너를 포함하는 최소 배포 단위 |
| Deployment | 복제본 수 및 롤링 업데이트 관리 |
| Service | Pod 집합에 대한 안정적 접근 지점 제공 |
| Ingress | 클러스터 외부 HTTP 트래픽 라우팅 |
| ConfigMap / Secret | 설정 및 민감 정보의 분리 관리 |
| Namespace | 리소스의 논리적 격리 |

이러한 추상화 계층과 YAML의 문법적 엄격성이 높은 학습 장벽을 형성한다. 반면 쿠버네티스 지식은 정형화되어 있고 매니페스트는 선언형 텍스트이므로, AI 에이전트가 요구사항 변환·장애 진단·변경안 제시를 수행하기에 적합하다.

---

## GitOps에서 GitAIOps로

쿠버네티스의 선언형 특성은 **GitOps**의 기반이 되었다. GitOps는 원하는 상태 전체를 Git에 저장하고(단일 진실 공급원), 모든 변경을 Pull Request로 수행하며, ArgoCD·Flux가 Git과 클러스터를 지속 동기화한다. 이로써 변경은 추적 가능해지고 롤백은 이전 커밋으로의 복귀로 성립한다.

그러나 GitOps는 선언된 상태의 적용은 해결하되, **무엇을 선언할 것인가**의 판단은 사람에게 남긴다. **AI 에이전트**가 이 공백을 보완한다. 요구사항으로부터 매니페스트 초안을 생성하고, 로그를 분석해 원인을 진단하며, 변경 근거를 자연어로 설명한다. AI는 GitOps를 대체하지 않고 그 **전단에 위치**하며, 본서는 이 결합을 **GitAIOps**로 정의한다.

```text
   [사람] 의도 → [AI] 탐색·비교·생성 → [Git] PR·리뷰·병합 → [Ops] 조정·적용·자기 치유
```

Git은 신뢰성과 추적성을, AI는 작성·진단의 효율성을, Ops는 실행의 안전성을 제공한다. 세 요소가 결합할 때 개발자는 세부 구현이 아니라 의도의 정의와 검토에 집중할 수 있다.

## 가드레일: 신뢰 가능한 위임의 조건

AI에게 인프라를 위임하는 것은 위험을 수반한다. LLM은 확률적으로 응답하며 환각(hallucination)을 일으킬 수 있고, 프로덕션 변경은 큰 파급 범위(blast radius)를 갖는다. **가드레일(guardrail)**은 이를 통제하는 장치이다.

- **CLAUDE.md**: 프로젝트 루트의 지침 파일로, 적용 전 diff 비교·승인, 민감 정보 비노출, Git PR을 통한 변경 등 규칙을 정의하여 에이전트의 동작 범위를 제약한다.
- **탐색 → 비교 → 실행**: 상태를 먼저 조회하고(비파괴적), 변경안을 diff·대안과 함께 제시하며, 사용자 승인 이후에만 적용한다. 읽기와 쓰기를 분리하고 그 사이에 사람의 판단을 개입시키는 것이 요체이다.

가드레일은 AI를 억제하는 듯 보이나, 실질적으로는 신뢰와 위임을 가능하게 하는 기제이다.
