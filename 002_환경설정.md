### 💬 Claude Code와 Copilot의 차이

책이 권하는 질문이다. 요지는 다음과 같다 — Copilot이 에디터 내에서 코드를 보완하는 도구라면, Claude Code는 **터미널에서 파일을 생성하고 명령을 실행하며 결과를 확인한 뒤 다음 행동을 결정하는 에이전트**다. 인프라 작업처럼 "실행 → 확인 → 수정" 반복이 필요한 작업에서 차이가 뚜렷하다.

### 💬 Artifact Registry와 Docker Hub의 차이

Docker Hub가 공개 중심의 범용 레지스트리라면, Artifact Registry는 GCP 프로젝트에 종속된 프라이빗 레지스트리다. GKE와 동일한 네트워크·IAM 체계 안에 있어 별도 pull secret 없이 이미지를 가져올 수 있으며, 리전을 서울로 지정하면 pull 속도 면에서도 유리하다.

## GKE 클러스터

**프롬프트**: `GKE 클러스터 생성해줘`

```bash
gcloud container clusters create notiflex-cluster \
  --zone=asia-northeast3-a \
  --machine-type=e2-medium \
  --num-nodes=2 \
  --spot \
  --gateway-api=standard \
  --disk-size=30
```

각 옵션은 비용과 이후 장을 고려한 결정이다:

| 옵션 | 이유 |
| ---- | ---- |
| Standard (Zonal) | Autopilot 대비 노드 직접 제어 가능 — 7장 노드풀 분리 실습에 필요. Zonal은 Regional 대비 관리 비용이 없음 |
| `--spot` | 온디맨드 대비 60~91% 저렴. 회수 가능성이 있으나 실습 환경에서는 허용 가능한 수준 |
| `--gateway-api=standard` | 5장 무중단 배포에서 필수. 누락 시 사후 활성화에 수 분 소요 |
| `--disk-size=30` | 기본 100GB를 30GB로 축소하여 디스크 비용 절감 |

### Go API 서버

`app/main.go` — 프레임워크 없이 표준 라이브러리만 사용한다:

```go
package main

import (
    "fmt"
    "log"
    "net/http"
    "os"
    "sync/atomic"
)

var counter int64

func healthHandler(w http.ResponseWriter, r *http.Request) {
    w.Header().Set("Content-Type", "application/json")
    w.Write([]byte(`{"status":"ok"}`))
}

func idHandler(w http.ResponseWriter, r *http.Request) {
    id := atomic.AddInt64(&counter, 1)
    pod, _ := os.Hostname()
    w.Header().Set("Content-Type", "application/json")
    fmt.Fprintf(w, `{"id":%d,"pod":"%s"}`, id, pod)
}

func main() {
    http.HandleFunc("/health", healthHandler)
    http.HandleFunc("/id", idHandler)
    log.Println("notiflex-api listening on :8080")
    log.Fatal(http.ListenAndServe(":8080", nil))
}
```

```text
// app/go.mod
module notiflex

go 1.25
```

`/id`가 Pod 이름을 함께 반환한다는 점이 중요하다. 어느 Pod이 응답했는지 확인할 수 있는 관측 장치 역할을 한다. Go 버전은 처음부터 **1.25**로 설정한다 — 6장의 valkey-go, 8장의 OTel SDK가 요구하는 버전이므로 사전에 맞춰두면 이후 버전 변경이 불필요하다.

### Dockerfile — scratch 멀티스테이지

```dockerfile
FROM golang:1.25-alpine AS build
WORKDIR /src
COPY go.mod ./
COPY . .
RUN CGO_ENABLED=0 go build -o /notiflex-api .

FROM scratch
COPY --from=build /notiflex-api /notiflex-api
EXPOSE 8080
ENTRYPOINT ["/notiflex-api"]
```

빌드 스테이지(golang:1.25-alpine, 수백 MB)와 실행 스테이지(scratch)를 분리한다. 최종 이미지는 정적 바이너리 하나로 **10MB 미만**이며, 셸과 패키지 매니저가 없어 공격 표면도 최소화된다.

### K8s 매니페스트

`k8s/smb/`에 namespace / deployment / service 세 파일을 생성한다. deployment의 핵심은 probe 설정이다:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: notiflex-api
  namespace: notiflex
spec:
  replicas: 2
  selector:
    matchLabels:
      app: notiflex-api
  template:
    metadata:
      labels:
        app: notiflex-api
    spec:
      containers:
        - name: api
          image: asia-northeast3-docker.pkg.dev/my-gitaiops-project/notiflex/api:v0.1.0
          ports:
            - containerPort: 8080
          readinessProbe:
            httpGet: { path: /health, port: 8080 }
            initialDelaySeconds: 3
          livenessProbe:
            httpGet: { path: /health, port: 8080 }
            initialDelaySeconds: 5
```

readiness probe는 트래픽 수신 준비 여부를, liveness probe는 프로세스 생존 여부를 판단한다. 애플리케이션에 `/health` 엔드포인트를 만든 이유가 여기에서 사용된다.

## /update-docs 커스텀 스킬

**프롬프트**: `/update-docs 커스텀 스킬 만들어줘`

`.claude/commands/update-docs.md` 파일 하나를 생성하면 Claude Code가 이를 슬래시 명령어로 자동 등록한다:

```markdown
---
description: 저장소 문서를 현재 작업 기준으로 갱신하고 변경 사항 커밋
---

# /update-docs

1. 이번 장 작업에서 변경된 내용을 파악한다 (Git 커밋 기록, 도입된 도구, 결정 사항)
2. JOURNEY.md, CLAUDE.md, docs/ 등 존재하는 문서를 갱신한다
3. 변경 사항을 커밋한다
```

이후 각 장이 끝날 때 `/update-docs` 입력만으로 문서 갱신과 커밋이 완료된다. 주목할 점은 **아직 존재하지 않는 문서까지 고려한 설계**라는 것이다. 5장에서 ADR, 6장에서 아키텍처 스냅샷이 추가되면 스킬 수정 없이 해당 문서들도 갱신 대상에 포함된다.


## 가드레일이라는 장치

이 책의 핵심 구조다. 독자의 자연어 입력 → 가드레일 문서(`prompt-guardrails/`) 매칭 → 검증된 절차로 실행 → 결과 템플릿(`result-templates/`)과 비교 검증.

2장에서도 여러 차례 작동했다. namespace race condition을 사전에 회피하고, Gateway API 옵션 누락을 방지하며, kubectl 컨텍스트를 강제한다. AI에게 작업을 전부 위임하는 것이 아니라 **검증된 절차 위에서 자율적으로 동작하게** 만드는 구조다.

기록 또한 이중 구조다. CLAUDE.md는 AI가 매 세션 참조하는 컨텍스트이고, JOURNEY.md는 진행 이력이다. 기록이 AI의 메모리가 되는 동시에, 이후 합류하는 인원에게는 온보딩 문서가 된다.


## 정리

| 단계 | 결과 |
| ---- | ---- |
| Claude Code | 동작 확인 + statusline, 가드레일 mode: auto |
| gcloud CLI | 575.0.0 설치, 서울 리전 + Artifact Registry 인증 |
| GitHub | notiflex-platform 저장소 + CLAUDE.md |
| GKE | Standard 클러스터 (e2-medium ×2, Spot, Gateway API 활성) |

직접 실행한 명령은 브라우저 인증 정도이며, 나머지는 프롬프트 한 줄씩으로 진행됐다. 환경 구성 단계임에도 배포 전략(Gateway API), 확장(노드풀), 관측(Pod 이름 반환) 등 이후 장을 위한 준비가 곳곳에 배치되어 있다. 다음 장에서는 ArgoCD로 GitOps 파이프라인을 구축한다.
